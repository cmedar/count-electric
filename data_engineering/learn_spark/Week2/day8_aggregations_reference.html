Interactive reference for Spark Day 8: aggregations, groupBy, countDistinct, pivot, HAVING equivalent, and shuffle cost

Oracle / MSSQL
Oracle SQL
SELECT region, product, SUM(revenue) AS total_rev, AVG(revenue) AS avg_rev, COUNT(*) AS cnt, MIN(revenue) AS min_rev, MAX(revenue) AS max_rev FROM orders GROUP BY region, product
Spark DataFrame API
from pyspark.sql import functions as F df.groupBy("region", "product") \ .agg( F.sum("revenue").alias("total_rev"), F.avg("revenue").alias("avg_rev"), F.count("*").alias("cnt"), F.min("revenue").alias("min_rev"), F.max("revenue").alias("max_rev") )
All agg functions run in one shuffle pass — no performance penalty for adding more expressions to .agg().
Spark SQL (identical to Oracle)
SELECT region, product, SUM(revenue), AVG(revenue), COUNT(*), MIN(revenue), MAX(revenue) FROM orders GROUP BY region, product
Exact vs. approximate
F.countDistinct("customer_id") F.approx_count_distinct("customer_id", rsd=0.05)
Exam trap: F.count(F.col("x").distinct()) does NOT work. Always use F.countDistinct() in the DataFrame API.
Oracle
COUNT(DISTINCT customer_id)
Spark DataFrame
F.countDistinct("customer_id")
approx_count_distinct uses HyperLogLog. ~2% error at rsd=0.05, much cheaper on large data — valid exam answer when precision isn't required.
Multi-column distinct
F.countDistinct("region", "product")
Counts distinct (region, product) pairs — equivalent to COUNT(DISTINCT region || product) in Oracle.
HAVING equivalent
Oracle SQL
SELECT region, SUM(revenue) AS total_rev FROM orders GROUP BY region HAVING SUM(revenue) > 100000
Spark DataFrame API
df.groupBy("region") \ .agg( F.sum("revenue").alias("total_rev") ) \ .filter(F.col("total_rev") > 100_000)
.filter() and .where() are identical in Spark — both compile to the same physical plan.
Combined WHERE + HAVING pattern
df.filter(F.col("status") == "completed") \ .groupBy("region") \ .agg(F.sum("revenue").alias("total_rev")) \ .filter(F.col("total_rev") > 100_000)
Pre-filter before groupBy to reduce shuffle input. Catalyst pushes it down anyway, but being explicit is cleaner.
Spark SQL — HAVING works natively
SELECT region, SUM(revenue) AS total_rev FROM orders WHERE status = 'completed' GROUP BY region HAVING SUM(revenue) > 100000
Basic pivot
df.groupBy("region") \ .pivot("quarter", ["Q1", "Q2", "Q3", "Q4"]) \ .agg(F.sum("revenue"))
Omitting the value list forces Spark to scan all distinct values first (extra job). Always supply the list in production.
Result schema
region | Q1 | Q2 | Q3 | Q4 -------+----------+----------+----------+-------- EMEA | 120000.0 | 95000.0 | 140000.0 | 180000.0
Multi-agg pivot — auto-suffixed columns
df.groupBy("region") \ .pivot("quarter", ["Q1", "Q2"]) \ .agg( F.sum("revenue").alias("sum"), F.avg("revenue").alias("avg") )
Produces columns: Q1_sum, Q1_avg, Q2_sum, Q2_avg
To unpivot (reverse a pivot), use stack() function or DataFrame.unpivot() (Spark 3.4+).
What happens during a groupBy shuffle
Stage 1 — local partial agg on each partition, then shuffle rows by key hash. Stage 2 — final merge per key on destination partitions.
200
Key config
spark.conf.set("spark.sql.shuffle.partitions", "200") spark.conf.set("spark.sql.adaptive.enabled", "true")
Default 200 shuffle partitions is too high for small data (creates tiny files), too low for large data. AQE coalesces empty/small partitions automatically when enabled.
