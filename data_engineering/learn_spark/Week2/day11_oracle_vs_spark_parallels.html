The key insight: in Oracle, complexity lives in JOIN tables. In Spark, it lives in nested columns. Day 11 is the toolkit for moving between these two worlds.
When does nested data appear?
Problem: customer has multiple phone numbers
Oracle — 2 tables + JOIN
customers id | name 1 | Ciprian customer_phones id | phone 1 | 0721000001 1 | 0721000002 1 | 0721000003 SELECT name, phone FROM customers c JOIN customer_phones p ON c.id = p.id
Spark — 1 row, array column
id | name | phones 1 | Ciprian | ["0721000001", "0721000002", "0721000003"] # Access first phone F.col("phones").getItem(0) # Has specific number? F.array_contains("phones", "0721000001") # One row per phone (= JOIN) F.explode("phones")
explode() = the Oracle JOIN result. collect_list() = the reverse, packing JOIN rows back into an array.
Going wide→long and back
wide→long: df.select("id", F.explode("phones").alias("phone")) long→wide: df.groupBy("id").agg(F.collect_list("phone").alias("phones"))
Problem: order has variable number of line items
Oracle — order_items table
orders order_id | customer 1001 | Ciprian order_items order_id | product | qty | price 1001 | Laptop | 1 | 4500 1001 | Mouse | 2 | 85 1001 | Keyboard | 1 | 220 -- Order total: SELECT order_id, SUM(qty * price) AS total FROM order_items GROUP BY order_id
Spark — array of structs
order_id | items 1001 | [ {product:Laptop, qty:1,price:4500}, {product:Mouse, qty:2,price:85}, {product:Keyboard,qty:1,price:220} ] # Order total — no explode needed F.aggregate("items", F.lit(0.0), lambda acc, x: acc + x.qty * x.price ).alias("total") # → 4875.0
Flatten to Oracle-style rows
df.select("order_id", F.explode("items").alias("item")) \ .select("order_id", F.col("item.product"), F.col("item.qty"), F.col("item.price"))
Enrich without flattening (Silver layer)
# Apply 10% discount to every item price df.withColumn("items", F.transform("items", lambda x: F.struct( x.product.alias("product"), x.qty.alias("qty"), (x.price * 0.9).alias("price") ) ) )
Rule of thumb: if you need the individual rows for analysis → explode (Silver table). If you're enriching/aggregating within the array → transform/aggregate (stay nested, cheaper).
Problem: different products have different attributes
Oracle — EAV antipattern
product_attributes prod_id | attr | value 101 | RAM | 16GB 101 | CPU | i7 102 | size | XL 102 | color | blue -- Get RAM for product 101: SELECT value FROM product_attributes WHERE prod_id=101 AND attr='RAM' -- Bloated, hard to query, -- no type safety
Spark — MAP column
prod_id | attrs 101 | {RAM:16GB, CPU:i7, GPU:RTX} 102 | {size:XL, color:blue} # Get RAM (null if missing) F.col("attrs")["RAM"] # What keys does each row have? F.map_keys("attrs") # Only rows with RAM key F.filter( F.map_contains_key(col,"RAM") ) # Add a new attribute F.map_concat("attrs", F.create_map( F.lit("source"), F.lit("catalog_v2") ) )
MAP column = Oracle EAV table collapsed into one cell. Keys can differ per row. Missing key returns null — no KeyError, no join miss.

The full Bronze → Silver → Gold pipeline using Day 11 tools
🟫 Bronze — land raw JSON as-is
from_jsonParse raw JSON string into typed struct
schema_of_jsonInfer schema from a sample (dev only)
→ One row per API call / Kafka message. Nested shape preserved.
↓
🥈 Silver — normalize to flat analytical rows
explodeArray → one row per element
col("struct.field")Access nested struct fields
map_keys / element_atUnpack map columns
→ Flat table. Looks like Oracle. One row per event/item/vehicle.
↓
🥇 Gold — aggregate into business-ready structures
collect_list / collect_setRe-pack rows into arrays (GROUP BY → array)
transform / filter / existsEnrich/filter within array columns
to_json / struct()Pack into nested output for APIs / dashboards
→ One row per entity (route, customer, product) with rich nested summary.
