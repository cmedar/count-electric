Python UDF — the basics

A UDF wraps arbitrary Python logic so Spark can call it per-row. Two syntaxes, same result:

from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

# Style 1 — decorator (preferred in notebooks)
@udf(returnType=StringType())
def clean_route(name: str) -> str:
    return name.strip().upper() if name else None

# Style 2 — explicit wrap
def _clean_route(name): ...
clean_route = udf(_clean_route, StringType())

df.withColumn("route_clean", clean_route("route_name"))

Return type is mandatory

Omitting it defaults to StringType(). Spark serialises/deserialises every value, so a wrong type causes runtime errors, not plan-time errors.

Python typeSpark type
strStringType()
intIntegerType() / LongType()
floatDoubleType()
boolBooleanType()
dict / NamedTupleStructType([...])
listArrayType(element_type)

Null handling

By default UDFs are called even when input is NULL. Your Python code must guard against None, or you get a NullPointerException at the executor.
@udf(returnType=StringType())
def safe_upper(s):
    return s.upper() if s is not None else None

# Or declare UDF as non-nullable to skip null rows entirely:
@udf(returnType=StringType(), useArrow=False)
def safe_upper2(s: str) -> str: ...
# But the UDF is still invoked — the declaration is metadata, not a guard
Tineretului–Română Transit use case: a UDF to normalise trip_headsign values (e.g. "Piața Romană" → "PIATA ROMANA") before joining with GTFS static data — except you'd actually reach for upper(regexp_replace(col, "ă|â|î|ș|ț", "a")) first to stay in Catalyst.

Pandas UDFs — three flavours

Pandas UDFs use Apache Arrow for serialisation: 10–100× faster than row-at-a-time Python UDFs. Three patterns cover different shapes of work.

1. Scalar (vectorised) — most common

Input: one pd.Series per column argument. Output: one pd.Series. Processes a batch of rows together.

from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf(StringType())
def normalise_headsign(s: pd.Series) -> pd.Series:
    return s.str.strip().str.upper().fillna("")

df.withColumn("headsign_norm", normalise_headsign("trip_headsign"))

2. Iterator of series — memory-efficient for large datasets

Streams batches via an iterator. Useful when you need to load a model/file once and reuse it across batches.

from typing import Iterator
from pyspark.sql.functions import pandas_udf

@pandas_udf(DoubleType())
def score_arrival(iterator: Iterator[pd.Series]) -> Iterator[pd.Series]:
    model = load_model()   # loaded once per partition
    for batch in iterator:
        yield model.predict(batch)

3. Grouped map — applyInPandas

Groups data like groupBy, sends each group as a full pd.DataFrame, returns a pd.DataFrame. Useful for per-group fitting, ranking, or custom aggregations not expressible in Spark SQL.

from pyspark.sql.types import StructType, StructField, ...

schema = StructType([
    StructField("route_id", StringType()),
    StructField("z_delay", DoubleType())
])

def z_score_delays(pdf: pd.DataFrame) -> pd.DataFrame:
    pdf["z_delay"] = (pdf["delay_sec"] - pdf["delay_sec"].mean()) / pdf["delay_sec"].std()
    return pdf[["route_id", "z_delay"]]

df.groupBy("route_id").applyInPandas(z_score_delays, schema=schema)
Gotcha: full group must fit in executor memory — if a single bus route has 50M rows, this will OOM.
Python UDF

Row-at-a-time. Pickle serialisation. Slowest. Simple to write.

Pandas UDF

Batch via Arrow. Fast. Null-safe by default (NaN in series). Needs pyarrow.

Why UDFs are opaque to Catalyst

Catalyst optimises Spark SQL via rule-based and cost-based rewrites: predicate pushdown, column pruning, join reordering, constant folding. It works by inspecting an expression tree made of known Spark expressions.

A UDF is a black box — Catalyst sees it as PythonUDF(…) with no knowledge of what's inside. This breaks three key optimisations:

OptimisationBuilt-inUDF
Predicate pushdownPushed to scan layerCannot push — evaluated after scan
Column pruningOnly reads needed columnsSafe only if input columns are explicit
Constant foldingEvaluated at plan timeAlways evaluated at runtime
Whole-stage codegenJVM bytecode, very fastBroken — switches to Python interpreter

Serialisation cost — the real bottleneck

Every row crossing the JVM → Python boundary is serialised (pickle or Arrow), sent over a socket, executed, then deserialised back. At scale this dominates.

Python UDF cost

Pickle serialise Socket round-trip Row-at-a-time

~100M rows/min on good hardware

Pandas UDF cost

Arrow serialise Batch boundary Vectorised ops

~1B rows/min — 10× improvement

Analogy (Oracle/MSSQL): a UDF is like a scalar function called per-row in a WHERE clause — Oracle can't push it into an index scan, forcing a full table scan. Built-in functions compile into the SQL engine; your function doesn't.

Practical impact in streaming

In Tineretului–Română Transit, a Python UDF on the arrival time window column would be called on every micro-batch row. A Pandas UDF processes the batch as a Series — far better. But if window() or date_trunc() cover the need, skip the UDF entirely.

Registering UDFs for SQL contexts

Python UDFs defined via @udf only work in the DataFrame API. To call them in SQL strings or notebooks via %sql, you must register with spark.udf.register.

# Register a Python UDF
spark.udf.register("clean_route", clean_route, StringType())

# Now callable in SQL
spark.sql("SELECT clean_route(route_name) FROM routes")

# Pandas UDFs also registerable the same way
spark.udf.register("normalise_headsign", normalise_headsign, StringType())

In Databricks Unity Catalog you can persist UDFs as catalog functions (CREATE FUNCTION catalog.schema.fn AS ...) so they survive cluster restarts. Python UDFs require a cluster with matching runtime.

DLT pipelines

In DLT (Delta Live Tables), define UDFs in a Python notebook cell above your @dlt.table definitions. DLT registers them in its execution context automatically.

# In DLT notebook
@udf(StringType())
def normalise_stop(s): return s.strip() if s else None

@dlt.table()
def silver_stops():
    return dlt.read("bronze_stops").withColumn("stop_norm", normalise_stop(col("stop_name")))

Decision rule: UDF vs built-in vs higher-order

Rule of thumb: a UDF is a last resort. Exhaust the function catalogue first.
ScenarioBest choiceWhy
String manipulation, date math, mathBuilt-inCatalyst-aware, JVM speed, no serialisation
Array / map transformation element-wiseHOF (transform, filter, aggregate)Catalyst-friendly; no Python overhead
Complex business logic, regex, external libsPandas UDFArrow batches, still vectorised
Per-group ML inference, custom aggregationPandas UDF grouped-mapFull DataFrame per group in Python
Simple one-off, low volume, prototypingPython UDFEasiest to write; acceptable at small scale
Truly unavoidable complex logic at scalePython UDF + cacheCache upstream to avoid repeated UDF calls

Flowchart (mental model)

When you reach for a UDF, ask in order:

1. Does a built-in Spark function do this? → Use it.
2. Is this a collection operation (per-element)? → Use transform / filter / aggregate.
3. Is throughput critical? → Pandas UDF (scalar or iterator).
4. Is it per-group logic? → applyInPandas.
5. None of the above → Python UDF, but measure the cost.

Count Electric example: computing EV range efficiency (range_km / battery_kwh) → pure arithmetic → use col("range_km") / col("battery_kwh"), not a UDF. Applying an ML model to predict range decay by temperature → Pandas UDF iterator pattern, load model once per partition.
