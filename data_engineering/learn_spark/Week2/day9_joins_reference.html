All join types with Oracle mapping
Semi and anti — the ones to know
left_semi — EXISTS equivalent
df_customers.join( df_orders, on="customer_id", how="left_semi" )
left_anti — NOT EXISTS equivalent
df_customers.join( df_orders, on="customer_id", how="left_anti" )
Semi/anti never duplicate rows from the left side — unlike an inner join on a non-unique key which multiplies rows.
Broadcast hint — explicit
from pyspark.sql.functions import broadcast df_orders.join( broadcast(df_products), on="product_id", how="inner" )
Spark SQL hint
SELECT /*+ BROADCAST(p) */ o.*, p.name FROM orders o JOIN products p ON o.product_id = p.id
Threshold config
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50mb") spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "-1")
Forcing broadcast on a table that's actually large will OOM the driver. Check the actual size first — Spark UI → Stages → Input Size.
AQE can promote a sort-merge join to broadcast at runtime if it discovers one side is small after partition pruning — even without a hint.
When to use broadcast
Click a strategy to see details
Broadcast hash join
fastest
Small side broadcast to all executors. Hash table built in memory. Large side streams through and probes. Zero shuffle. Requires one side < autoBroadcastJoinThreshold or explicit hint. OOM risk if overused on large tables.
Shuffle hash join
medium
Both sides shuffled by key. Smaller side builds a hash table per partition, larger side probes it. Faster than sort-merge when one side fits in executor memory but is too large to broadcast to all executors. Hint: SHUFFLE_HASH.
Sort-merge join
default large
Both sides shuffled and sorted on join key, then merged in a single pass. Handles arbitrarily large data — neither side needs to fit in memory. Default when both sides are large. Hint: MERGE. Two shuffles = highest cost.
Shuffle replicate NL
avoid
Used for cross joins and non-equi joins with no usable key. One side replicated to all partitions, nested-loop join applied. Cost grows as O(n×m). Hint: SHUFFLE_REPLICATE_NL. Only intentional cross joins should reach this strategy.
Duplicate column names
joined = df_orders.join(df_customers, df_orders.customer_id == df_customers.customer_id) joined.select("customer_id") joined = df_orders.join(df_customers, on="customer_id") joined = df_orders.alias("o").join( df_customers.alias("c"), F.col("o.customer_id") == F.col("c.customer_id") ).select("o.customer_id", "c.name")
NULL key matching
df.join(df2, df["key"].eqNullSafe(df2["key"]), how="inner" )
Standard equi-join: NULL ≠ NULL (same as Oracle). Use eqNullSafe when you want NULL = NULL semantics.
Row multiplication on non-unique key
orders (10M rows) inner join events (50M rows) on customer_id (non-unique both sides) → potentially billions of output rows
Verify join key uniqueness before joining large tables. Use left_semi when you only need existence, not the joined columns.
Skew on join key
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true") spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
AQE splits skewed partitions at runtime. For extreme skew on a known key, combine with salting or isolate the hot key.
