Aggregations

groupBy().agg() — Core Functions

from pyspark.sql import functions as F

df.groupBy("dept", "region").agg(
    F.count("*").alias("row_count"),
    F.sum("salary").alias("total_sal"),
    F.avg("salary").alias("avg_sal"),
    F.min("salary").alias("min_sal"),
    F.max("salary").alias("max_sal"),
    F.countDistinct("role").alias("unique_roles"),
)

collect_list and collect_set

# collect_list — preserves order of insertion, KEEPS duplicates
df.groupBy("dept").agg(
    F.collect_list("name").alias("all_names")   # ["Alice","Alice","Bob"]
)

# collect_set — DEDUPLICATES values, result order is NOT guaranteed
df.groupBy("dept").agg(
    F.collect_set("name").alias("unique_names")  # ["Alice","Bob"] — unordered
)
Exam trap: collect_list preserves duplicates and retains insertion order. collect_set removes duplicates and returns an unordered set. Do not assume sorted output.

pivot() — Rows to Columns

# Pivot: make category values into column headers
df.groupBy("dept").pivot("quarter").sum("revenue")
# → columns: dept, Q1, Q2, Q3, Q4

# Provide an explicit values list for better performance
# (avoids a full scan to discover distinct values)
df.groupBy("dept").pivot("quarter", ["Q1","Q2","Q3","Q4"]).sum("revenue")
Exam trap: Calling pivot() without specifying the values list forces Spark to do an extra pass over the entire column to find distinct values — expensive on large datasets. Always supply the list when you know it.

rollup() and cube()

# rollup — left-to-right hierarchical subtotals
# groupings: (year, quarter, region), (year, quarter), (year), ()
df.rollup("year", "quarter", "region").agg(F.sum("sales"))

# cube — ALL possible grouping combinations
# groupings: every subset of (year, quarter, region) including ()
df.cube("year", "quarter", "region").agg(F.sum("sales"))

# NULL in result = "all values" for that dimension (the rollup/cube row)
# Use grouping() to distinguish NULL-key rows from actual NULL data
df.rollup("dept", "region").agg(
    F.sum("salary"),
    F.grouping("dept").alias("is_total_dept")  # 1 = subtotal row
)

groupBy vs rollup vs cube

Function What it produces Grouping combinations (3 cols) Use case
groupBy(a,b,c) One row per unique (a, b, c) combination 1 — only (a,b,c) Standard aggregation
rollup(a,b,c) Subtotals from left: (a,b,c), (a,b), (a), () 4 — n+1 combos Hierarchical drill-down (year → quarter → month)
cube(a,b,c) Every subset including grand total 8 — 2ⁿ combos OLAP-style cross-tab reports
Remember: In rollup/cube results, a NULL in a grouping column means "all values" (the subtotal). It does NOT mean the underlying data had a null key.

Window Functions

Defining a WindowSpec

from pyspark.sql import Window
from pyspark.sql import functions as F

# Basic spec: partition + order
w = Window.partitionBy("dept").orderBy(F.col("salary").desc())

# With explicit row frame
w_rows = (Window.partitionBy("dept")
                .orderBy("date")
                .rowsBetween(Window.unboundedPreceding, Window.currentRow))

# With range frame (uses value-based bounds, not row counts)
w_range = (Window.partitionBy("dept")
                 .orderBy("salary")
                 .rangeBetween(-1000, 1000))

df.withColumn("rank", F.rank().over(w))

Ranking Functions

rank()
Leaves gaps after ties.
Ties at 1st → next is 3rd.
Example: 1, 1, 3, 4, 4, 6
dense_rank()
No gaps after ties.
Ties at 1st → next is 2nd.
Example: 1, 1, 2, 3, 3, 4
row_number()
Always unique, sequential.
Ties broken arbitrarily.
Example: 1, 2, 3, 4, 5, 6
percent_rank()
Relative rank as a fraction 0.0–1.0.
Formula: (rank - 1) / (total rows - 1)
# All three ranking functions — classic exam comparison
w = Window.partitionBy("dept").orderBy("salary")

df.withColumn("rnk",        F.rank().over(w)) \
  .withColumn("dense_rnk",  F.dense_rank().over(w)) \
  .withColumn("row_num",     F.row_number().over(w))

Navigation Functions

# lead — look FORWARD n rows (default 1); null if no row exists
df.withColumn("next_salary",  F.lead("salary", 1).over(w))
df.withColumn("next2_salary", F.lead("salary", 2, 0).over(w))  # default=0

# lag — look BACKWARD n rows (default 1); null if no row exists
df.withColumn("prev_salary", F.lag("salary", 1).over(w))

# first_value / last_value over window
df.withColumn("first_sal", F.first("salary").over(w))
df.withColumn("last_sal",  F.last("salary").over(w_rows))
Watch out: last_value / F.last() with the default frame (RANGE up to current row) often gives unexpected results. Use an explicit rowsBetween(unboundedPreceding, unboundedFollowing) frame to get the true last value in the partition.

Aggregate Functions Over a Window

# Running total (cumulative sum) — ORDER BY triggers default RANGE frame
w_cum = Window.partitionBy("dept").orderBy("date")
df.withColumn("running_total", F.sum("sales").over(w_cum))

# Full partition sum (no ORDER BY = whole partition as frame)
w_full = Window.partitionBy("dept")
df.withColumn("dept_total", F.sum("sales").over(w_full))

# Running avg, count
df.withColumn("running_avg", F.avg("sales").over(w_cum))
df.withColumn("running_cnt", F.count("*").over(w_cum))

ntile and QUALIFY

# ntile(n) — divides partition into n equal buckets (1-based)
w = Window.partitionBy("dept").orderBy("salary")
df.withColumn("quartile", F.ntile(4).over(w))   # 1, 2, 3, or 4

# QUALIFY in Spark SQL — filter on window result without subquery
spark.sql("""
    SELECT dept, name, salary,
           RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rnk
    FROM employees
    QUALIFY rnk <= 3
""")

Default Frame Rules

WindowSpec has ORDER BY?Default frameEffect
Yes RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW Running/cumulative calculation. Ties included in the "current row" range.
No RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING Whole partition — every row sees the same aggregate value.
Key insight: Adding ORDER BY to a window spec changes the frame from "whole partition" to "up to current row". This is why SUM OVER without ORDER BY gives the total, but with ORDER BY gives a running total.

Semi and Anti Joins — SQL Equivalents

# LEFT SEMI — equivalent to EXISTS / IN
orders.join(customers, on="customer_id", how="left_semi")
# Returns only orders where a matching customer exists
# Result has ONLY orders columns — no customer columns are added

# LEFT ANTI — equivalent to NOT EXISTS / NOT IN
orders.join(customers, on="customer_id", how="left_anti")
# Returns orders with NO matching customer — orphaned records
# Result has ONLY orders columns
Left Semi = EXISTS
Filters left side to rows that have a match. Only left-side columns in output. Like a filter, not a join — no column duplication.
Left Anti = NOT EXISTS
Keeps left rows that have no match. Perfect for finding orphaned records, unprocessed events, or data quality gaps.

Null Key Behaviour

Critical: null != null in joins. Two rows with null in the join key will never match each other. This applies to all standard join types. If you need null-safe equality, use df1.col.eqNullSafe(df2.col) as the join condition. 
# Null-safe join — nulls match nulls
df1.join(df2, df1["key"].eqNullSafe(df2["key"]), "inner")

String Join vs Condition Join — Column Deduplication

# String join — keeps ONE "id" column (deduplicates)
result = df.join(other, "id")               # or on=["id","dept"]
# result.columns → [..., "id", ...]  — single id column

# Condition join — keeps BOTH "id" columns, must drop manually
result = df.join(other, df["id"] == other["id"])
# result.columns → [..., "id", ..., "id", ...]  — duplicate!
result = result.drop(other["id"])            # must disambiguate
Best practice: Prefer string-column join syntax when both DataFrames share the same column name. Use condition syntax when you need to join on expressions or columns with different names.

Broadcast Join

from pyspark.sql.functions import broadcast

# Force broadcast of a small DataFrame
large_df.join(broadcast(small_df), "id")

# Spark auto-broadcasts when table is below threshold
spark.conf.get("spark.sql.autoBroadcastJoinThreshold")  # default 10MB

# Disable auto-broadcast
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)

Skewed Joins and AQE

Adaptive Query Execution (AQE) — Spark 3.x: AQE automatically detects skewed partitions during a sort-merge join and splits the large partition into smaller pieces, then replicates the matching data from the other side. No manual repartition needed.

Delta Lake

Core Properties

ACID Transactions
Atomicity, Consistency, Isolation, Durability. Concurrent reads and writes are safe. No partial writes — either the whole operation commits or nothing changes.
Time Travel
Every write creates a new version in the transaction log. You can query any past version by version number or timestamp. Default retention: 30 days.
Schema Enforcement
Delta rejects writes that don't match the table schema (schema-on-write). Use mergeSchema or overwriteSchema for intentional schema evolution.
Change Data Feed (CDF)
Tracks row-level changes (inserts, updates, deletes) — enables incremental processing pipelines. Must be enabled: delta.enableChangeDataFeed = true.

DESCRIBE HISTORY

-- View full version history
DESCRIBE HISTORY employees;

-- Columns: version, timestamp, operation, operationParameters, userMetadata
-- Example operations: WRITE, UPDATE, DELETE, MERGE, OPTIMIZE, VACUUM END

Time Travel — Querying Past Versions

-- SQL: by version number
SELECT * FROM employees VERSION AS OF 3;

-- SQL: by timestamp
SELECT * FROM employees TIMESTAMP AS OF '2024-01-15';
SELECT * FROM employees TIMESTAMP AS OF '2024-01-15 08:30:00';
# Python: by version
df = spark.read.format("delta").option("versionAsOf", 3).load(path)

# Python: by timestamp
df = spark.read.format("delta").option("timestampAsOf", "2024-01-15").load(path)
Exam trap: Time travel only works within the data retention window. After running VACUUM, old data files are deleted and older versions become inaccessible even though the transaction log still references them.

OPTIMIZE and ZORDER BY

-- Compact small files into larger ones (reduces S3 list overhead)
OPTIMIZE employees;

-- ZORDER: co-locate related data for data skipping on filter queries
OPTIMIZE employees ZORDER BY (dept, hire_date);

-- After optimize, query: "WHERE dept = 'Eng'" skips most files entirely
OPTIMIZE
Compacts many small files (from frequent small writes) into larger Parquet files. Improves read performance. Does not delete old files — use VACUUM for that.
ZORDER BY
Re-orders data within files by the specified column(s) to maximise data skipping. Only helps queries that filter on the Z-ordered column(s). Best for high-cardinality filter columns.

VACUUM

-- Delete data files older than retention period (default 7 days / 168 hours)
VACUUM employees;

-- Specify a custom retention
VACUUM employees RETAIN 30 HOURS;

-- DANGEROUS: delete everything except the current version
-- Disables the safety check first
spark.conf.set("spark.databricks.delta.retentionDurationCheck.enabled", "false")
VACUUM employees RETAIN 0 HOURS;
CommandWhat it does to old filesAffects time travel?
OPTIMIZE Creates new compacted files; marks old small files as removed in log Old files kept until VACUUM runs
VACUUM Physically deletes data files beyond retention period from storage Breaks time travel for deleted versions

MERGE INTO (Upsert)

MERGE INTO target t
USING source s ON t.id = s.id
WHEN MATCHED THEN
  UPDATE SET t.name = s.name, t.salary = s.salary
WHEN NOT MATCHED THEN
  INSERT (id, name, salary) VALUES (s.id, s.name, s.salary)
WHEN NOT MATCHED BY SOURCE THEN   -- Spark 3.4+ / Databricks
  DELETE;
# Python equivalent using DeltaTable
from delta.tables import DeltaTable

target = DeltaTable.forPath(spark, "/path/to/target")
target.alias("t").merge(
    source.alias("s"), "t.id = s.id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

UDFs & Caching

Defining and Using a Python UDF

from pyspark.sql.functions import udf
from pyspark.sql.types import StringType, IntegerType

# Method 1: decorator
@udf(returnType=StringType())
def clean_name(name):
    if name is None: return None
    return name.strip().title()

df.withColumn("clean", clean_name(F.col("name")))

# Method 2: wrap an existing function
def multiply(x, y): return x * y
mul_udf = udf(multiply, IntegerType())

# Method 3: register for use in spark.sql() strings
spark.udf.register("clean_name", clean_name, StringType())
spark.sql("SELECT clean_name(name) FROM employees")

UDF Pitfalls

Serialization overhead
Each row is serialized from JVM (Java) to Python and back. For large DataFrames, this is a significant bottleneck. Prefer built-in functions (F.*) whenever possible.
No Catalyst optimization
Spark treats UDFs as black boxes — it cannot push predicates through or reorder them. The optimizer sees a UDF call and stops optimizing around it.
Null handling
UDFs receive None for null values by default. If your function doesn't handle None, it will raise an exception at runtime. Always add null checks.
Return type required
You must declare the return type. Without it Spark defaults to StringType() which may cause runtime errors or silent type coercions.

Pandas UDFs (Vectorized UDFs)

from pyspark.sql.functions import pandas_udf
from pyspark.sql.types import DoubleType
import pandas as pd

# SCALAR — Series in, Series out (most common)
@pandas_udf(returnType=DoubleType())
def normalize(s: pd.Series) -> pd.Series:
    return (s - s.mean()) / s.std()

df.withColumn("norm_salary", normalize(F.col("salary")))
Pandas UDF typeInput → OutputUse case
SCALAR Series → Series Column-level transformation (default type for @pandas_udf)
GROUPED_MAP DataFrame → DataFrame Complex group-level computation, arbitrary output schema per group
GROUPED_AGG Series → scalar Custom aggregation function over a group
Why Pandas UDFs are faster: They use Apache Arrow to transfer data between JVM and Python in batches (columnar format), instead of row-by-row Python object serialization. Typically 10–100x faster than row-at-a-time Python UDFs. Requires pyarrow to be installed.

Top Exam Traps

  • Semi join keeps ONLY left columns. It acts like a filter, not a merge — you never get columns from the right-side DataFrame in a semi or anti join result.
  • Anti join = "no match" rows. Use left_anti to find orphaned records, unprocessed events, or data quality violations (rows in table A with no corresponding row in table B).
  • String join deduplicates the join key column. df.join(other, "id") → one "id" column. Condition join df.id == other.id → two "id" columns, must drop manually.
  • VACUUM default retention = 7 days (168 hours). Time travel only works for versions whose underlying data files have not been vacuumed away.
  • OPTIMIZE compacts files; it does NOT delete them. VACUUM physically deletes old data files. These are two separate operations — OPTIMIZE alone does not free storage.
  • ZORDER only helps queries that filter on the Z-ordered column(s). It does nothing for queries that scan all rows or filter on unrelated columns.
  • collect_list preserves duplicates; collect_set deduplicates. collect_set output order is non-deterministic.
  • pivot() without a values list does an extra full scan. Always provide the values list when you know the distinct values upfront.
  • cache() is lazy — first action triggers materialization. Calling cache() alone does nothing until an action (count, show, write) is triggered.
  • Pandas UDF requires pyarrow. It's vectorized (batch), not row-by-row, so it's significantly faster than a regular Python UDF.
  • AQE handles skewed joins automatically in Spark 3.x. No manual repartition needed for skewed sort-merge joins when AQE is enabled (default in Databricks).
  • null != null in join conditions. Null keys never match. Use eqNullSafe() if you need null-safe equality in a join.
  • Delta time travel: integer → VERSION AS OF; date string → TIMESTAMP AS OF.
  • Window ORDER BY changes the default frame. Without ORDER BY: whole partition. With ORDER BY: running (unbounded preceding to current row). This is why SUM OVER behaves differently with vs without ORDER BY.
  • rollup() is hierarchical left-to-right. rollup(a, b, c) produces n+1 = 4 grouping combinations. cube() produces 2ⁿ = 8.
  • regexp_extract returns "" on no match — not null. Filtering with .isNull() will not catch unmatched rows. Use F.length(col) > 0.
  • datediff(end, start) — end comes first. Reversing the arguments gives a negative result. The most commonly swapped date function on the exam.
  • dayofweek: 1 = Sunday, 2 = Monday, ..., 7 = Saturday. Spark does not follow Monday = 1 conventions used in many databases.
  • element_at is 1-indexed; getItem is 0-indexed. element_at(col, 1) and getItem(0) both return the first element. element_at(col, 2) and getItem(2) return different elements.
  • size(null) = -1; cardinality(null) = 0. Both exclude null arrays when filtering > 0, but the raw return values differ.
  • explode drops null/empty arrays; explode_outer keeps them as a null row. If row count is unexpectedly low after explode, this is almost always the cause.
  • array_position returns 0 (not null) when the value is not found. Checking isNull() will never match — filter on > 0 instead.
  • from_json silently returns null on malformed JSON — no exception. Always filter or route bad rows to a dead-letter table after parsing.
  • map_concat — right map wins on key conflict. map_concat(a, b): if both have key "status", b's value is kept.
