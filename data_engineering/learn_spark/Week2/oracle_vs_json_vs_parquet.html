Three formats — what they are and when you see them
🗄 Oracle / MSSQL table
Format: row-oriented heap storage
Nested types: not supported — 1NF enforced
Human readable: via SQL only
Analytics speed: slow at scale — reads all columns
Where you see it: OLTP systems, legacy pipelines
Complexity lives in: JOIN relationships between tables
📄 JSON (lines / API response)
Format: text, row-oriented
Nested types: native — arrays and objects built-in
Human readable: yes — open in any text editor
Analytics speed: slow — no compression, full parse each row
Where you see it: APIs, Kafka messages, Bronze landing zone
Complexity lives in: nested objects inside the document
⚡ Parquet file
Format: binary, columnar
Nested types: native — array, map, struct in spec
Human readable: no — binary, needs Spark/tool to read
Analytics speed: fast — column pruning + compression
Where you see it: Silver/Gold Delta tables, data lake files
Complexity lives in: nested columns, already typed
Key difference: do you need from_json?
Parquet already decoded the types — nested columns arrive as Array/Struct/Map objects ready to use. JSON arrives as a raw string — you must call from_json first to get the same result.
Same data — three physical representations
Oracle — normalized, 4 rows across 2 tables
orders order_items order_id|customer order_id|product |qty|price 1001 |Ciprian 1001 |Laptop |1 |4500 1001 |Mouse |2 |85 1001 |Keyboard|1 |220
JSON — one document, nested array (Bronze layer)
{ "order_id": 1001, "customer": "Ciprian", "items": [ {"product": "Laptop", "qty": 1, "price": 4500}, {"product": "Mouse", "qty": 2, "price": 85}, {"product": "Keyboard", "qty": 1, "price": 220} ] } # Arrives as raw STRING in Spark — needs from_json to become typed
Parquet — one row, nested array, already typed (Silver/Gold)
df.printSchema() # root # |-- order_id: integer ← already an INT, not a string # |-- customer: string # |-- items: array # | |-- element: struct # | | |-- product: string ← already typed # | | |-- qty: integer ← already typed # | | |-- price: double ← already typed # No from_json needed — just use it directly: df.select("order_id", F.explode("items").alias("item")) \ .select("order_id", F.col("item.product"), F.col("item.qty"), F.col("item.price"))
Why columnar matters for analytics
Oracle / JSON — row scan
SELECT SUM(price) FROM orders -- Must read every column -- of every row to find price: [1001,Ciprian,Laptop,1,4500] [1001,Ciprian,Mouse,2,85] [1001,Ciprian,Keyboard,1,220] -- Wastes I/O on order_id, -- customer, product, qty
Parquet — column scan
SELECT SUM(items.price) -- Reads ONLY the price -- column off disk: price: [4500, 85, 220, 800...] -- order_id, customer, product, -- qty never touched -- 4-10x faster at scale
Format flow in a real Databricks pipeline
Decision: JSON string column vs Parquet nested column
The Day 11 toolkit splits naturally: from_json / to_json are for the Bronze edge (JSON strings in, JSON strings out). explode / transform / collect_list / struct are for Silver↔Gold work on already-typed Parquet columns.
Delta = Parquet + transaction log
Delta table on disk: /mnt/silver/orders/ ├── _delta_log/ ← transaction log (JSON files) │ ├── 00000.json ← "added these Parquet files" │ ├── 00001.json ← "updated these rows" │ └── 00002.json ← "deleted these rows" ├── part-00000.parquet ← actual data, columnar, nested-ready ├── part-00001.parquet └── part-00002.parquet # The data files ARE Parquet. # Delta just wraps them with ACID semantics.
When you write a Delta table and read it back, nested columns (arrays, maps, structs) are preserved exactly — no re-parsing. This is why Bronze→Silver writes nested Parquet/Delta and Silver→Gold reads it back as typed objects.
Schema evolution with nested types in Delta
# New API version adds a field to the items struct # Delta can merge the new schema automatically: df.write \ .format("delta") \ .option("mergeSchema", "true") \ .mode("append") \ .save("/mnt/silver/orders") # New field appears as null in old rows — no rewrite needed
Reading back — no from_json, types intact
df = spark.read.format("delta").load("/mnt/silver/orders") # items column is Array<Struct> — already typed # Just explode and work with it directly df.select(F.explode("items").alias("i")).select("i.product","i.price")
